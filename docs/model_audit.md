## 1. 予測対象

| 項目 | 内容 |
|------|------|
| タスク | 3クラス分類: ホーム勝利 / 引き分け / アウェー勝利 |
| 出力 | 各クラスの確率 (%) + 予想スコア + 信頼度 |
| 最終判定 | argmax(home_win_prob, draw_prob, away_win_prob) |

---

## 2. 予測パイプライン構造

```
順位表スクレイプ → 特徴量スコア化 → 重み付き合算 → raw_home_advantage
                                                        ↓
                            ┌───────────────────────────┤
                            │                           │
                    [Gemini API あり]           [Gemini API なし]
                            │                           │
                    プロンプトに全データ渡し     advantage_to_probs()
                    LLMが最終確率を決定         シグモイド変換で確率算出
```

**重要**: Gemini APIがある場合、重み付きスコアはLLMへの「参考情報」に過ぎず、
最終確率はLLMが自由に決定する。つまり重み最適化の効果はGeminiモード時に限定的。

## 4. 未来情報リーク分析

### 4.1 直接リークなし (現時点)
現在の設計では、予測時に使用するデータは全て「試合前」の情報:
- 順位表: 当該試合前の累積勝点
- フォーム: 当該試合前の直近結果
- xG: シーズン途中の累積統計

### 4.2 バックテスト時のリスク (要注意)

| リスク | 内容 | 危険度 |
|--------|------|--------|
| 順位表リーク | バックテストでシーズン最終順位表を使うと未来リーク | **高** |
| xGリーク | FBrefのシーズン累積xGを全試合に適用すると未来含む | **高** |
| 資本力スコア | 静的DBなので時間依存なし→リスク低 | 低 |
| フォーム | 直近5試合を正しく時系列で取得すれば安全 | 中 |
| H2H | 過去対戦のみ参照→安全 | 低 |

**結論**: 現時点の「リアルタイム予測」では未来リークなし。
ただしバックテスト基盤を作る際に、時系列分割を厳密にしないと容易にリークする。

## 5. 重みベース設計の限界

### 5.1 構造的限界
1. **線形結合の限界**: `raw_advantage = Σ(weight_i × (home_score_i - away_score_i))` は特徴量間の交互作用を捉えられない
2. **シグモイド変換の硬直性**: `tanh(x*3)*0.30` の形状が固定→引き分けの表現力が低い
3. **ベース確率の固定**: home=40%, draw=25%, away=35% がハードコード → データから学習されていない

### 5.2 スコア化関数の問題
- `score_team_strength`: 勝点を相対比率に変換 → シーズン序盤に不安定(総勝点が小さい)
- `score_recent_form`: 重み `[0.35, 0.25, 0.20, 0.12, 0.08]` が手動設定
- `score_home_advantage`: **完全固定値** (0.62) → チーム/スタジアムごとの差異なし
- `score_attack_rate` / `score_defense_rate`: シュリンク係数 `games/10` が理論的裏付けなし

### 5.3 Gemini依存のブラックボックス性
- Geminiモード時: 重み付きスコアは「参考」でありLLMが最終判断
- 再現性なし: 同じ入力でもGeminiの出力が毎回変わる (temperature=0.25)
- バックテスト不可: 過去試合をGeminiに再投入しても当時の判断は再現不能

## 8. 改善の方向性 (概要)

### 短期 (データ量制約内)
1. **死にパラメータ削除**: 15.3%の無駄な重みを有効パラメータに再配分
2. **引き分け確率モデル改善**: 固定ベース(25%)→実力接近度からの動的算出
3. **シグモイド変換のパラメータ調整**: `tanh(x*K)*M` のK, Mを探索
4. **フォーム期間の探索**: 5試合固定→3-8試合の範囲で比較

### 中期 (バックテスト基盤構築後)
5. **ELOレーティング導入**: 試合結果から動的に更新する戦力指標
6. **重み最適化**: Optunaによる体系的探索
7. **ホームアドバンテージの動的化**: チーム別/スタジアム別の補正

### 長期 (十分なデータ蓄積後)
8. **ロジスティック回帰/序数回帰への移行**: 重み付き合算の限界を超える
9. **Geminiの役割見直し**: 定量モデルの補正役に限定

---

## 9. 結論

現行モデルは「手動重み × 線形結合 × シグモイドまたはLLM判断」という構造で、
再現性・検証可能性に大きな課題がある。特に:

- **22パラメータ中6つ(15.3%)が常にデフォルト値で死んでいる**
- **Geminiモード時の予測は再現不可能**
- **バックテスト基盤が存在しない**
- **19件のサンプルでは統計的評価が不可能**

統計モデル(`advantage_to_probs`)のみをバックテスト対象とし、
過去シーズンデータを用いた時系列検証を構築することが最優先。
