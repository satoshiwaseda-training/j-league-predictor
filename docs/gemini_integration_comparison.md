> 作成日: 2026-04-09

## 1. 評価方針

Gemini APIは非決定論的(temperature>0)かつコストがかかるため、
大規模バックテストは不可能。以下の方式で評価する:

- **統計モデル(v7 refined)**: バックテストで定量評価済み (val=2025, n=377)
- **Gemini既存モード**: 定量評価不可 (再現性なし)
- **Gemini + v7事前確率**: 定量評価不可 → 安全性と情報正確性で検証

## 3. Gemini統合で渡される情報の検証

### 3.1 事前確率の正確性
テストケース:
- 接近試合 (raw_adv=0.05, closeness=0.85): H=37% D=33% A=30% → draw注意あり
- 一方的試合 (raw_adv=0.25, closeness=0.25): H=54% D=22% A=24% → draw注意なし
- ELOなし (raw_adv=0.0, closeness=1.0): H=33% D=36% A=31% → draw注意あり

**検証結果**: 事前確率は3ロジットsoftmaxから正しく算出されている。

### 3.2 ELO要約の正確性
- ELO期待勝率はcalculate_parameter_contributions内のeloパラメータから取得
- elo_home_score/elo_away_scoreが正しく渡されていればOK
- ELO未取得時: eloパラメータがない → ELO行は非表示

### 3.3 draw注意シグナルの発動条件
- closeness >= 0.5 かつ d_pct >= 25%
- val=2025の377試合中、概算で約100-150試合でシグナル発動見込み
- draw実際発生率25.7%と整合的

## 4. Gemini統合の期待効果

### 4.1 positive効果
1. **draw見落とし防止**: 既存Geminiはdraw_probを低めに設定する傾向
   → 統計モデルのdraw注意シグナルで修正される可能性
2. **一貫性向上**: 統計モデルの事前確率を出発点にすることで、
   Geminiの「気分」に依存しない安定した予測の底上げ
3. **情報補完**: Geminiが持つ定性知識(選手コンディション、戦術変化)と
   統計モデルの定量分析が相互補完

### 4.2 リスク
1. **過度な依存**: Geminiが統計モデルの確率をそのまま返す可能性
   → 「参考値」「出発点」表現で軽減
2. **プロンプト長増加**: 約100-150文字追加 → タイムアウトリスクは低い
3. **暴走**: 統計モデルの確率を無視して極端な予測を出す可能性は残る
   → これは統合前から同じ問題

## 6. 本番確認手順

Gemini統合の実際の効果は、本番環境で以下を確認:
1. 個別予測タブでGemini予測を実行
2. `reasoning`フィールドに統計モデル事前確率への言及があるか確認
3. draw注意シグナル表示時にdraw_probが適切に反映されているか確認
4. 予測確率の合計が100%になっているか確認
