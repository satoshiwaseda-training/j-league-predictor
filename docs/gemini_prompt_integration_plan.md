## 1. 追加すべきELO関連情報

### プロンプトへの追加セクション案

```
## ELO レーティング
{home_team}: ELO {home_elo:.0f} (リーグ平均1500)
{away_team}: ELO {away_elo:.0f}
ELO差: {elo_diff:+.0f} (正=ホーム有利)
ELO期待勝率: ホーム{home_elo_pct:.0f}% / アウェイ{away_elo_pct:.0f}%
```

### データフローの依存点
- `app.py: get_elo_scores(division, home, away)` → (elo_h, elo_a)
- `app.py: cached_elo_ratings(division)` → ratings dict
- ELO生値は `ratings.get(home, 1500)` で取得可能

## 3. 既存プロンプトとの整合性

### 現行 predict_with_gemini() のプロンプト構造
1. 試合情報 (チーム名、順位、勝点)
2. 攻撃・守備力 (Dixon-Coles)
3. コンディション指標 (試合間隔、規律、損耗率、資本力)
4. 重み付きパラメータ貢献度
5. 直近フォーム
6. H2H
7. 天気
8. 出力形式指示

### 追加位置
- ELOセクション: 「攻撃・守備力」の後、「コンディション指標」の前
- 統計モデル事前確率: 「重み付きパラメータ貢献度」の後

### 注意点
- Geminiへの情報過多を避ける (プロンプトが長すぎると精度低下)
- ELO数値は参考情報として渡し、Geminiの判断を拘束しない
- 事前確率はGeminiの「出発点」として提示し、補正を促す
